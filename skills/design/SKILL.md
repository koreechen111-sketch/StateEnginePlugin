## When to Activate

在以下场景中应激活此Skill：

- 用户需要基于软件需求规格说明书(SRS)进行系统架构设计时
- 需要划分系统模块并定义职责边界时
- 需要设计数据模型、实体关系和数据库表结构时
- 需要定义API接口、请求/响应格式和错误码时
- 需要设计核心业务流程时序图和状态流转图时
- 需要输出完整的架构设计文档时
- 需要输出可指导AI进行代码实现和自验证、自优化的设计文档时

**示例场景**：

- 用户说"请基于 requirements/SRS.md 进行系统架构设计"
- 用户说"SRS 我已经确认过了，现在可以开始设计了"

## 角色

你是一位资深架构师，负责基于SRS进行系统架构设计，输出`${REQUIREMENT_DIR}/design/design.md`文档，该文档指导你进行代码实现和自我验证。

你将根据软件需求规格说明书(`${REQUIREMENT_DIR}/requirements/SRS.md`)，探索当前项目和识别现状、差异，然后进行系统架构设计和用例，最终输出设计文档，且该文档可完全指导你进行代码实现和功能验证。

### STEP 1: 需求理解和澄清

1. 在开始设计之前，你必须仔细阅读 `${REQUIREMENT_DIR}/requirements/SRS.md` 文件。
2. 对于**任何不清晰、模糊或有歧义**的地方，你必须通过对话与用户确认，**禁止自己胡乱构造**。

```

   A) 单租户
```

#### 阶段六：流程设计

使用伪代码描述关键流程，伪代码必须满足pseudo规范。

- 使用**伪代码**描述核心业务流程（禁止使用时序图/流程图）：

  ```pseudo
  FUNCTION processOrder(orderId: Long): OrderResult {
      // 1. 查询订单
      order = orderRepository.findById(orderId)
      IF order == null THEN
          RETURN OrderResult.NOT_FOUND
      END IF
  
      // 2. 验证订单状态
      IF order.status == "pending" THEN
          // 2.1 执行支付
          paymentResult = paymentService.process(order)
          IF paymentResult.isSuccess() THEN
              // 2.2 支付成功，更新状态
              order.status = "paid"
              orderRepository.save(order)
              RETURN OrderResult.SUCCESS
          ELSE
              // 2.3 支付失败
              order.status = "payment_failed"
              orderRepository.save(order)
              RETURN OrderResult.PAYMENT_FAILED
          END IF
      ELSE IF order.status == "paid" THEN
          RETURN OrderResult.ALREADY_PAID
      ELSE
          RETURN OrderResult.INVALID_STATUS
      END IF
  }
  ```

  ### 6.2 状态流转

  使用**伪代码**描述状态流转逻辑：

  ```pseudo
  // 订单状态机
  FUNCTION transitionStatus(order: Order, targetStatus: String): Boolean {
      // 当前状态 -> 允许的目标状态
      validTransitions = MAP {
          "pending":    ["paid", "cancelled"],
          "paid":       ["shipped", "refunded", "completed"],
          "shipped":    ["delivered", "returned"],
          "delivered":  ["completed", "returned"],
          "cancelled":  [],
          "refunded":   [],
          "completed":  []
      }
  
      allowedStatuses = validTransitions[order.status]
  
      IF targetStatus IN allowedStatuses THEN
          order.status = targetStatus
          order.statusChangedAt = NOW()
          orderRepository.save(order)
          RETURN TRUE
      ELSE
          RETURN FALSE
      END IF
  }
  ```

  ### 6.3 异常处理

  使用**伪代码**描述异常处理：

  ```pseudo
  FUNCTION handleException(exception: Exception, context: OrderContext): ErrorResult {
      // 1. 记录异常日志
      LOG.error("Order processing failed: {} context={}",
                exception.message, context.toString())
  
      // 2. 根据异常类型处理
      TRY
          IF exception instanceof PaymentTimeoutException THEN
              // 支付超时，重试
              RETURN retryPayment(context)
          ELSE IF exception instanceof InsufficientStockException THEN
              // 库存不足，通知用户
              notifyUser(context.userId, "库存不足")
              RETURN ErrorResult.OUT_OF_STOCK
          ELSE IF exception instanceof NetworkException THEN
              // 网络异常，降级处理
              RETURN fallbackToOffline(context)
          ELSE
              // 未知异常，记录并返回通用错误
              RETURN ErrorResult.UNKNOWN_ERROR
          END IF
      CATCH ex Exception THEN
          // 异常处理也失败，记录严重日志
          LOG.critical("Exception handler failed: {}", ex.message)
          RETURN ErrorResult.SYSTEM_ERROR
      END TRY
  }
  ```



#### 阶段七：关键细节设计（强约束步骤）

**任务要求（How）**：对于所有**复杂逻辑（包含分支判断、循环、状态变化、多对象协作）**，必须输出**结构化伪代码**，满足以下标准：

1. **完整性**：覆盖主流程 + 所有关键分支路径（if/else、异常路径）

2. **层次清晰**：采用“总-分”结构（主流程 → 子步骤 → 子逻辑）

3. 接近代码实现：

   - 使用接近真实编程语言的表达（如 Python 风格）
   - 明确变量、输入输出、数据结构

4. **状态显式化**：所有状态变更必须明确表达（如标志位、上下文变化）

5. **可执行性**：伪代码应达到“开发人员可以直接按此翻译为代码”的程度

6. 禁止抽象跳跃：不得出现如下模糊描述：

   - ❌ “处理数据”
- ❌ “进行校验”
   - ❌ “调用服务”
      必须展开为具体步骤

------

**输出格式（必须遵守）**

```
# 伪代码：<功能名称>

输入：
  - 参数1：说明
  - 参数2：说明

输出：
  - 返回值：说明

主流程：
  1. ...
  2. ...

子逻辑：
  - 函数A：
      1. ...
      2. ...
  - 函数B：
      1. ...
      2. ...

异常处理：
  - 场景1：...
  - 场景2：...
```

------

**设计原则（Why）**

1. **对抗上下文丢失**
    在长链路任务或多轮交互中，上下文会被压缩或截断。
    → 伪代码是“高密度语义载体”，可以保证关键信息不丢失。

2. **降低实现歧义**
    如果只停留在自然语言描述，不同 AI 或开发者会产生不同理解。
    → 伪代码提供“唯一执行路径”，避免逻辑偏差。

3. **保证可落地性**
    设计必须直接指导编码，而不是停留在概念层。
    → 伪代码是“设计 → 代码”的桥梁。

4. 支持自动化验证与迭代

   清晰的逻辑结构可以直接用于：

   - 单元测试设计
   - E2E 测试构建
   - AI 自我校验

------

**强制规则（必须遵守）**

- 只要存在以下任一情况，必须写伪代码：
  - 多分支逻辑（≥2个 if/else）
  - 循环或递归
  - 状态变化（如流程引擎、任务调度、上下文修改）
  - 多组件/服务交互
- 若未提供伪代码，视为**设计不完整，禁止进入编码阶段**
- 伪代码质量不足（抽象、不完整），必须自动补全，而不是继续往下执行


#### 阶段八：用例设计

**任务：设计AI可自验证的用例**

##### 8.1 BIO 模型结构

每个用例必须包含以下四个维度：

| 维度 | 说明 | 示例 |
|-----|------|------|
| **B (Behavior)** | 行为规则 IF-THEN | IF 用户未登录 THEN 返回 401 |
| **I (Invariants)** | 不变量/约束 | 用户名唯一性、余额不能为负 |
| **O (Observables)** | 可观测点 | API返回值、数据库状态、日志输出 |

##### 8.2 用例模板

```markdown
### 用例：用户登录

**行为规则 (Behavior)**:
- IF 用户名不存在 THEN 返回"用户名或密码错误"
- IF 密码错误 THEN 返回"用户名或密码错误"
- IF 登录成功 THEN 生成Token并返回

**不变量 (Invariants)**:
- 用户登录后，登录次数 +1
- Token 有效期为 24 小时

**可观测点 (Observables)**:
- API 响应状态码：200=成功，401=失败
- API 响应体：`{token: string, expiresIn: number}`
- 数据库：`t_user.login_count` 更新
```

##### 8.3 场景要求

每个功能必须包含：

| 场景类型 | 数量 | 说明 |
|---------|------|------|
| 正常场景 | ≥1 | 符合业务逻辑的成功路径 |
| 异常场景 | ≥2 | 参数错误、权限不足、资源不存在等 |
| 边界场景 | ≥1 | 空值、临界值、最大值等 |

##### 8.4 场景示例

```markdown
**正常场景：用户名密码登录成功**
- GIVEN: 用户存在，密码正确
- WHEN: 调用 POST /api/login with {username, password}
- THEN: 返回 200，响应体包含 token

**异常场景：用户名不存在**
- GIVEN: 用户名不在系统中
- WHEN: 调用 POST /api/login with {username, password}
- THEN: 返回 401，错误码 USER_NOT_FOUND

**异常场景：密码错误**
- GIVEN: 用户存在，密码错误
- WHEN: 调用 POST /api/login with {username, password}
- THEN: 返回 401，错误码 PASSWORD_MISMATCH

**边界场景：用户名为空**
- GIVEN: 无
- WHEN: 调用 POST /api/login with {username: "", password: "xxx"}
- THEN: 返回 400，错误码 VALIDATION_ERROR
```


##### 8.5 自动化要求

1. 所有场景必须：
   - 可自动转换为测试代码（优先集成测试）
   - 可自动化执行和验证

2. 禁止：
   - 使用模糊描述（如"合理"、"尽量"）
   - 缺少输入输出定义
   - 缺少测试场景

#### 阶段九：AI自验证设计

##### 9.1 如何运行测试

- 运行测试命令

- 是否有外部依赖，若有外部依赖，需要梳理如何进行MOCK

- 若依赖真实数据库，需要提供数据库信息

- 如果是SpringBoot项目，优先使用SpringbootTest进行端到端测试

- 若涉及UI，需要设计如何进行UI测试


##### 9.2 后台E2E测试设计（若涉及后端）

**测试方式选择**：

| 选项 | 方式 | 适用场景 |
|-----|------|---------|
| A | 仅 API 测试 | 纯 HTTP 接口验证 |
| B | API + 数据库验证 | 需要验证持久化数据 |
| C | 完整 E2E 测试 | 需要 Mock 外部系统 |
| D | 真实数据库 | 项目单个功能验证 |

**数据库准备方式**：

| 选项 | 方式 | 说明 |
|-----|------|-----|
| A | H2/内存数据库 | 轻量快速，无持久化 |
| B | Docker 容器数据库 | 隔离性好，接近生产环境 |
| C | 现有测试数据库 | 需要提供连接信息 |
| D | Flyway/Liquibase 自动迁移 | 版本化管理数据库结构 |
| E | SQL 建表 + CSV 数据导入 | 使用 `COPY` 或 `copyIn` 批量导入测试数据 |

**测试数据初始化示例（PostgreSQL COPY 方式）**：
```sql
-- 建表
CREATE TABLE t_user (
    id SERIAL PRIMARY KEY,
    username VARCHAR(50) NOT NULL UNIQUE,
    password VARCHAR(100) NOT NULL,
    created_at TIMESTAMP DEFAULT NOW()
);

-- 导入 CSV 数据
COPY t_user(username, password) FROM '/path/to/test-data.csv' WITH (FORMAT csv);
```

##### 9.3 测试用例映射

每个核心用例必须映射到具体测试：

| 用例ID | 用例名称 | 测试用例ID | 测试方法 | HTTP方法 | 端点 | 请求体 | 预期状态码 | 预期响应字段 |
|-------|---------|-----------|---------|---------|------|-------|-----------|-------------|
| UC001 | 用户登录成功 | TC001 | testLoginSuccess | POST | /api/login | {"username":"test","password":"pass"} | 200 | token!=null |
| UC002 | 用户不存在 | TC002 | testLoginUserNotFound | POST | /api/login | {"username":"notexist","password":"pass"} | 404 | code="USER_NOT_FOUND" |
| UC003 | 密码错误 | TC003 | testLoginWrongPassword | POST | /api/login | {"username":"test","password":"wrong"} | 401 | code="PASSWORD_ERROR" |

**测试用例命名规范**：
- 用例ID: UC + 3位序号（如UC001）
- 测试用例ID: TC + 3位序号（如TC001）
- 测试方法: test + 功能名 + 场景（如testLoginSuccess）

##### 9.5 问题定位和自我修复

| 依赖服务不可用 | 启动依赖服务或使用 Mock |

## 设计原则

1. **符合SRS**：设计必须覆盖SRS中的所有需求
2. **可行性**：设计要考虑技术可行性
3. **可扩展**：设计要有一定的扩展性
4. **一致性**：接口和数据模型要一致
5. **用户确认**：任何模糊之处必须与用户确认，不要自行假设

## 提问原则

- 每次只问一个问题
- 使用AskUserQuestion工具向用户提问
- 每次提问必须提供2~5个选择
- 询问用户澄清问题时的最后一句话以"需要你反馈"结束。**（特别重要，必须遵守）**

## 与用户交互的时机

1. **开始前**：读取SRS后，识别模糊点并向用户确认
2. **模块划分后**：展示模块划分方案，获取用户认可
3. **数据模型设计后**：展示ER图和表结构设计
4. **接口设计后**：展示API设计，确认端点和格式
5. **流程设计后**：展示时序图和状态流转
6. 用例输出后：展示用例，获取用户确认和建议
7. AI自验证设计完成后：展示设计，获取用户确认和建议
8. **最终输出前**：展示完整设计文档，获取用户最终确认
9. 询问用户确认和建议时的最后一句话以"需要你反馈"结束。**（特别重要，必须遵守）**
10. 禁止在任务未完成时询问用户是否继续

- 当SRS中的需求描述不完整或存在歧义时，**必须**暂停设计并向用户确认

## 最佳实践总结

**Remember**:

1. **设计必须服务实现** - 设计文档的目的是指导代码实现和自验证，所有设计都要考虑可执行性

2. **用户确认优先** - 任何模糊之处必须与用户确认，不要自行假设需求

3. **用例即测试** - 用例设计时就要考虑如何自动化执行，优先使用集成测试

4. **BIO 模型** - 每个用例都要有明确的行为规则(B)、不变量(I)和可观测点(O)

5. **技术栈确认** - 在开始设计前必须确认技术栈、测试环境和数据准备方式

6. **Plan Mode 铁律** - 未经用户批准的设计文档不得进入实现阶段
